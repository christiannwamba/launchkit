## Core Thread

Every app, every product, every piece of software ever built comes down to five jobs:

1. **Store** the data somewhere
2. **Secure** it so the right people access it
3. **Process** it into something useful
4. **Transmit** it so it gets where it needs to go
5. **Present** it so people can interact with it

That's it. That is the first principle. Once someone understands this, every tool stops feeling mysterious and starts feeling like a job assignment:

* **Convex** stores the data
* **WorkOS** secures it
* **Our code** processes it
* **Convex + Vercel** transmit it
* **Next.js** presents it

The entire video is about making this mental model click, then proving it works by shipping a real product.

## Outline

### 1. Cold Open: Show the Finished Product

Open on the live customer feedback board: logged in, users submitting feedback, clean interface, deployed and running. Then say the line:

> "This was built with ChatGPT, Next.js, Convex, and WorkOS. We went from zero to a live app. And it cost about $20."

### 2. The First Principle

Before touching any tool, land the idea that every app is just five jobs: store the data, secure it, process it, transmit it, present it. Use an analogy. Think of a package delivery system: the warehouse stores the package, security verifies who can access it, the sorting center processes where it should go, the delivery network transmits it, and the doorstep is where it is presented to the recipient. Software is no different.

Then map the stack directly onto those five jobs. This is the moment the audience stops seeing a pile of tools and starts seeing a system.

### 3. Tension: Why Beginners Fail Before They Ship

Most first-time builders don't fail because coding is too hard. They fail because there are too many decisions. Too many tools, too many tutorials, unclear auth, unclear backend, deployment confusion, expensive AI app builders hiding the real system.

Frame the core problem: beginners don't need infinite options. They need **one path that removes decisions**.

### 4. The Beginner-Proof Stack

Introduce the stack as the answer to the tension. Keep it short and decisive.

* **Next.js** for the presentation layer
* **Convex** for the data layer
* **WorkOS** for the security layer (bundled through Convex to reduce friction)
* **Our code** for the processing layer
* **GitHub** for source control
* **Vercel** for deployment

This stack is not the only way. It's the fewest-decisions path. That's what makes it right for someone starting from scratch.

### 5. Why a Customer Feedback Board

Explain why this is the demo: it feels like a real product, it could be used by founders or small teams, it has obvious business value, and it's simple enough for a first-time builder. It gives real product surfaces: public-facing UI, auth, a data model, a submission flow, and deployment.

### 6. Phase 1: Setup

Show setup as a checklist, not a wall of steps. Create the Next.js app, connect Convex, enable WorkOS auth through the Convex path, initialize Git, prepare the GitHub repo, connect environment variables.

Keep translating for the viewer: "This gives us the presentation layer." "This handles where our data lives." "This gives users sign-in without us building auth from scratch."

This is where trust gets built.

### 7. Phase 2: Prompting and Generating the App

Show the master prompt structure: what the app is, what pages it needs, what schema it needs, how users interact, design constraints, what to avoid.

Walk through what gets generated: the feedback submission flow, list and detail views, the auth-aware experience, basic polish.

Do not present prompting like magic. Present it like direction. The point of the prompt is to reduce ambiguity. The stack makes the output easier to wire up. Include a moment where the result is imperfect and you tighten it. That makes the process honest.

### 8. Phase 3: Shipping

Push to GitHub. Connect the repo to Vercel. Add environment variables. Deploy. Verify auth works in production. Show the live URL.

This section should feel like crossing the finish line. The real value is not "the AI wrote code." It's that the repo exists, the deployment exists, the app is live, and someone could use it today.

### 9. The Bigger Insight

The breakthrough is not AI by itself. It's combining AI with an opinionated stack that maps cleanly onto the first principle: store, secure, process, transmit, present. Once someone can ship one small real product, they stop seeing software as something only other people make.

### 10. Closing and CTA

You don't need to build the next giant startup first. Build one useful thing. Keep the stack narrow. Keep the scope small. Ship before you optimize.

CTA: comment with what app they want to ship next, or mention a follow-up on domain setup, improving the product, or adding payments.
